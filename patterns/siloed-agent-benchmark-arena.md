## Pattern

Evaluate coding agents by giving each agent a copied task workspace, letting it modify only that workspace, then scoring the resulting artifacts with a centralized evaluator that the agent does not control.

The pattern is especially useful for benchmarks where agent output must be measured by objective external behavior: compile success, tests, correctness checks, performance, cost, latency, or artifact quality.

## Problem

Agent benchmarks get noisy when:

- Agents can mutate shared state across tasks.
- Agents write or influence their own score files.
- Baseline measurements are stale or external.
- Runs lack reproducible logs and artifacts.
- Expensive hardware jobs need parallelism but not a whole control-plane service.
- Agents overfit visible tests without being checked on held-out cases.

## Forces

- Agents need enough autonomy to do real work.
- Evaluators need independence from the agent.
- Benchmark runs need replayable evidence.
- Parallel workers need to avoid stepping on each other.
- Hardware-specific benchmarks need local measurement, not abstract scoring.
- Credentials and task commands are powerful, so the environment must be treated as trusted lab infrastructure.

## Shape

1. Keep a versioned task corpus.
2. For each run, create a timestamped run directory.
3. For each task, copy the task into an isolated workspace.
4. Measure baseline behavior before agent edits.
5. Launch the agent inside the task workspace.
6. Materialize any helper files needed by the evaluator.
7. Run centralized compile/test/performance checks after the agent exits.
8. Write structured result files from the evaluator, not from the agent.
9. Aggregate results into run-level summaries.
10. Preserve logs, prompts, outputs, and artifacts for audit.

## Optional Extensions

### Multi-Worker Descriptor Queue

For multi-GPU or multi-machine labs, represent each task as a descriptor file and move descriptors through queue directories:

- `pending`
- `running`
- `done`
- `failed`

Workers claim tasks with an atomic file move. Each worker gets a device mask and a local workspace/home/cache. This keeps scheduling inspectable without requiring a database or service.

### Held-Out Generalization Pass

After visible-task optimization, copy completed workspaces into a second evaluation pass and inject unseen test cases. Compare original and optimized code again on the held-out cases.

This catches agents that overfit known inputs or benchmark scaffolding.

### Validator Agent

Use a separate validator pass to inspect new task definitions before admitting them into the corpus. The validator can check required files, command shape, baseline behavior, expected outputs, and documentation quality.

The validator should report findings, but final corpus admission should still be policy-controlled.

## Why It Works

The main separation is:

- Agents mutate code.
- The benchmark owns truth.

That makes results easier to trust. The agent can be creative, but scoring stays deterministic, centralized, and auditable.

## Implementation Notes

- Store all task metadata in structured files.
- Keep score files generated by the evaluator.
- Log the exact agent command, model/config, start/end time, and exit status.
- Record baseline and optimized measurements separately.
- Make run suffixes/path fragments safe with a strict character allowlist.
- Treat task commands as trusted code unless the runner has a real sandbox.
- Keep agent credentials scoped and isolated from personal machine state.

## Good Fit

- GPU kernel optimization benchmarks
- SWE-agent A/B tests
- Prompt, skill, MCP, or tool integration trials
- Compiler/codegen evals
- Browser or CLI automation evals with external checkers
- Any benchmark where agent outputs can be judged by independent commands

## Poor Fit

- Open submissions from untrusted users without sandboxing
- Benchmarks where evaluator logic must be subjective or agent-mediated
- Casual local use on machines with sensitive credentials
- Leaderboards without published methodology and hardware controls

## Reusable Checklist

- Is each task copied into a clean workspace?
- Is baseline behavior measured locally before agent edits?
- Is the agent prevented from writing final score files?
- Are evaluator commands independent of agent output text?
- Are logs and artifacts preserved per task?
- Can results be aggregated without rereading agent transcripts?
- Are held-out tests available for overfitting checks?
- Are credentials and privileged devices isolated to benchmark hosts?
